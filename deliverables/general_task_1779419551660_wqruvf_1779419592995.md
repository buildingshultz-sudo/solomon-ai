# Persistence test task

To ensure that “this should survive restart,” treat it as a requirement for persistence across system restarts. Here’s a complete plan you can follow:

---

### 1. Define What Must Persist
- **Scope**: Identify the data, configuration, and state that must remain intact after a restart (e.g., files, database entries, service states, environment variables).
- **Granularity**: Decide whether you need to persist only critical data, user sessions, application settings, or entire system state.
- **Tolerance for loss**: Clarify acceptable recovery point (RPO) and recovery time (RTO). This guides how often you sync data and how quickly it must be available.

---

### 2. Choose Persistence Mechanisms
- **File-based**: Use files stored on durable disk (e.g., writing JSON/CSV/SQLite in a known directory). Ensure the path is on persistent storage, not temporary directories.
- **Databases**: For structured data, leverage databases with built-in durability (transactions, journaling).
- **State stores**: For caches or in-memory state that must be persistent, consider tools such as Redis with AOF (Append-Only File) persistence or a message queue with durable storage.
- **Configuration**: Store configs in source-controlled files or centralized configuration management (e.g., Consul, etcd) rather than memory.
- **Environment variables**: If values must survive restart, write them to configuration files or secret managers so they’re reapplied on startup.

---

### 3. Implement Persistence Logic
- **Write-through**: When state changes, immediately write to the persistent medium. Avoid keeping critical data solely in RAM.
- **Checkpoints**: If continuous writes are too heavy, use periodic checkpoints (e.g., every N minutes or on major milestones).
- **Graceful shutdown hooks**: Capture final state using system signals (`SIGTERM`/`SIGINT`) to trigger saving state before shutdown.
- **Initialization**: On startup, load the persisted state and validate it (checksum, schema checks) before applying.

---

### 4. Ensure Durability
- **Atomic writes**: Use temporary files + rename or database transactions to prevent partial writes.
- **Backups**: Schedule backups to separate storage, especially for critical data. Store metadata checksums to detect corruption.
- **Replication**: For high availability, replicate data to multiple nodes or use managed services with multi-zone durability.

---

### 5. Automate Recovery
- **Startup scripts**: Automate loading state/config on boot so no manual intervention is required.
- **Self-tests**: After loading state, run consistency checks and log results.
- **Monitoring**: Instrument logs and metrics to alert if state fails to load, files are missing, or data is corrupted.

---

### 6. Test the Restart Process
1. **Baseline Save**: Write known state, confirm it’s stored (e.g., inspect file, query DB).
2. **Restart Simulation**: Restart the service/system (or kill and relaunch in development).
3. **Verification**: On restart, validate that the state matches the baseline (automated tests plus manual spot checks).
4. **Chaos Testing**: Simulate power loss or abrupt crashes to ensure durability mechanisms (journaling, write-ahead logs) handle it.

---

### 7. Document and Operationalize
- **Playbooks**: Document exact steps for saving, backing up, and restoring state.
- **Versioning**: Track schema versions of stored data to handle migrations.
- **Security**: Protect persisted data at rest (encryption) and manage access control.

---

Following this workflow ensures the required state truly survives restarts reliably, with monitoring and testing to prove it.
