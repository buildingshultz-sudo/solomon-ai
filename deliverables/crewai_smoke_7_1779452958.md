# PC Agent test: list running processes

# **Executive Summary**

This report documents the PC Agent command-path validation exercise performed on Jed Shultz’s Windows workstation via the relay architecture. The objective was to execute a PowerShell instruction to enumerate all running Chrome processes and capture their metadata (Process ID, CPU utilization, start time). Despite multiple submission attempts, the relay consistently returned a `HTTP 400` error citing a “Missing command,” indicating a structural issue in the command payload being sent to the agent queue. This document analyzes the interaction chain, identifies the technical root cause, and lays out a corrective action plan so future remote instrumentation commands execute successfully. Key next steps include aligning with the v3.1 relay API contract, upgrading the PC Agent to v5 for enhanced command schemas, and running a smoke-test workflow that includes automated validation of command payload structure before queue submission.

| 2              | `{"command_type":"powershell","payload":"command=Get-Process chrome | Select-Object ... "}`                         | `HTTP 400 {"ok":false,"error":"Missing command"}`     | Failed |
| 3              | `{"command_type":"powershell","payload":"Get-Process"}`                                                               | `HTTP 400 {"ok":false,"error":"Missing command"}`     | Failed |

### 3.1 Observations
- Each payload used a `payload` field, aligning with legacy v4 command formatting.
- The relay explicitly anticipates a `command` field (per v3.1 spec).
- No command ever reached the PC Agent; failure occurred at request validation at the relay layer.

### 3.2 Root Cause Hypothesis
**Misaligned JSON schema**: The current `send_pc_agent_command` helper likely expects `{ "command_type": "...", "command": "..." }`. Using `payload` instead of `command` leads to the “Missing command” rejection. This interpretation fits both the response message and recent relay updates emphasizing compatibility with both v4 and v5 agents.

## **4. Technical Analysis**

### 4.1 Relay Schema Requirements
- **Minimum fields**:
  - `command_type` (e.g., `powershell`, `chrome_url`)
  - `command` (actual shell instruction)
- **Optional fields**:
  - `timeout`, `tab_id`, etc., depending on command type.

Historical logs suggest older implementations used `payload`, but post-v3.1 patch documentation (May 22, 2026) indicates the consolidated `command` key. Failure to update the helper or memory resulted in the current mismatch.

### 4.2 Agent Version Importance
PC Agent remains on v4.0.0. Upgrading to v5 adds:
- Self-upgrade handler
- Tab management
- Auto-screenshot capture
- Better error messaging
- Improved schema validation

While the immediate issue is schema mismatch at the relay, upgrading the agent is a best practice so future features (tab enforcement, screenshot confirmation) work reliably.

### 4.3 Process Listing Command
Target command:

```powershell
Get-Process chrome | Select-Object Id, ProcessName, CPU, StartTime
```

- Returns process ID, friendly name, cumulative CPU seconds, and start timestamp.
- Output is essential for verifying Chrome’s health and ensuring max-five-tab policy enforcement.
- Could also be piped to `ConvertTo-Json` for structured returns if needed.

## **6. Recommendations**

### 6.1 Correct Command Schema (Immediate)
- Update helper to send:
  ```json
  {
    "command_type": "powershell",
    "command": "Get-Process chrome | Select-Object Id, ProcessName, CPU, StartTime"
  }
  ```
- Re-run test after schema fix to confirm success.

### 6.2 PC Agent Upgrade to v5 (Next 24 hours)
- Use `/agent/upgrade` endpoint with latest installer.
- Benefits: schema auto-validation, improved logging, auto screenshot, tab enforcement synergy.

### 6.3 Automated Preflight Validator
- Implement a lightweight validator script on the VPS that checks outgoing payloads against the relay schema before submission.
- Prevents future manual errors and accelerates debugging.

### 6.4 Command Result Logging
- Once command execution succeeds, pipe results into structured log stored under `reports/pc-agent/2026-05-23-process-check.md`.
- Attach screenshot or JSON for recordkeeping.

### 6.5 Broader Diagnostics Suite
- After schema fix, run:
  1. `Get-Process chrome`
  2. `Get-Process | Sort-Object CPU -Descending | Select -First 5`
  3. `Get-Process | Measure-Object`
- Captures CPU hogs, verifies system load, ensures stable baseline for editing sessions.
